# BLE 调试页设计说明

## 背景

当前项目已经具备 BLE 连接、发现服务与特征、发送数据、监听通知的底层能力，但缺少一个面向开发和联调的可视化调试入口。由于模拟器无法可靠调试 BLE，真机联调需要一个能直接针对“当前设备详情页对应设备”进行消息收发和日志观察的页面。

本次设计目标是在不改动 BLE 核心连接链路的前提下，为设备详情页增加一个稳定、可复用、正式包可见的调试入口。

## 目标

- 在设备详情页增加一个常驻 `BLE 调试` 按钮。
- 点击按钮后进入独立调试页，调试页只针对当前设备详情页这台设备。
- 调试页不负责重新连接设备，而是复用当前 `BleService` 中已有的连接会话。
- 支持手动发送消息，并在页面内自动持续接收设备回包。
- 支持查看当前设备的 `serviceUUID` 和 `charUUID`，并手动选择发送和监听通道。
- 支持查看结构化日志，便于定位协议、特征值选择和回包格式问题。

## 非目标

- 不新增蓝牙自动重连逻辑。
- 不改动 `lib/ble-service.ts` 中现有连接时序、断开时序和错误分类策略。
- 不修改原生配置、权限声明或 BLE 插件配置。
- 不尝试自动推断业务协议，也不为具体设备协议做解析适配。
- 不把调试页做成隐藏入口，正式包内保持可见。

## 用户路径

1. 用户已通过现有流程连接设备并进入设备详情页。
2. 用户在设备详情页点击 `BLE 调试` 按钮。
3. App 跳转到独立 BLE 调试页，并携带当前设备 `id` 与 `name`。
4. 调试页读取 `BleService.getConnectedDevice()`：
   - 如果当前连接设备存在且与详情页设备一致，则允许调试。
   - 如果不存在，或存在但不是当前设备，则展示错误提示并禁用发送/监听操作。
5. 调试页展示当前设备所有可用的 service 和 characteristic。
6. 用户选择发送通道、监听通道，输入消息并发送。
7. 页面自动接收设备通过 notify/indicate 回传的数据，并写入日志区。
8. 用户返回设备详情页时，仅停止本页面建立的监听，不主动断开设备。

## 信息架构

调试页划分为四个区域：

### 1. 设备状态区

- 显示当前设备名称与设备 ID。
- 显示当前 `BleService` 中连接设备的 ID。
- 当连接状态不匹配时，显示明确的不可操作提示。

### 2. 通道选择区

- 拉取并展示当前连接设备的所有 services。
- 展示每个 characteristic 的 UUID 以及能力标记：
  - `writeWithResponse`
  - `writeWithoutResponse`
  - `notify`
  - `indicate`
  - `read`
- 支持选择一个发送 characteristic。
- 支持选择一个监听 characteristic。

### 3. 发送区

- 文本输入框，用于输入要发送的消息。
- `withResponse` 开关，用于选择写入模式。
- `发送` 按钮，点击后调用 `BleService.send(...)`。

### 4. 日志区

- 展示按时间倒序或顺序追加的调试日志。
- 日志条目包含：
  - 时间
  - 方向：`SEND` / `RECV` / `INFO` / `ERROR`
  - 关联的 `serviceUUID / charUUID`
  - 原始 Hex
  - 尝试解码后的 UTF-8 文本
- 提供 `清空日志` 按钮。

## 路由与文件变更

### 新增页面

- `app/(device)/ble-debug.tsx`

职责：

- 承载当前设备的 BLE 调试 UI。
- 展示服务与特征值。
- 执行手动发送。
- 启动和停止当前页面的监听。
- 展示调试日志。

### 修改页面

- `app/(device)/[id].tsx`

职责变化：

- 在设备详情页增加一个进入 `BLE 调试` 页的常驻按钮。
- 跳转时携带当前设备 `id` 与 `name`。

### 修改路由布局

- `app/(device)/_layout.tsx`

职责变化：

- 注册 `ble-debug` 页面。

### 文案处理

- 按用户最新要求，本次不改 `i18n`。
- BLE 调试入口和调试页文案直接以内联中文实现。

## 数据流

### 页面初始化

1. 调试页从路由参数中读取 `deviceId` 与 `deviceName`。
2. 调用 `BleService.getConnectedDevice()` 获取当前连接设备。
3. 校验当前连接设备是否与详情页设备一致。
4. 若一致，则调用原生设备对象上的 `services()` 和 `characteristics()` 获取服务与特征值清单。

### 发送数据

1. 用户手动选择发送用的 `serviceUUID` 和 `charUUID`。
2. 用户输入消息内容。
3. 页面调用 `BleService.send(serviceUUID, charUUID, message, withResponse)`。
4. 页面写入一条 `SEND` 日志。
5. 如失败，写入一条 `ERROR` 日志。

### 自动接收数据

1. 用户手动选择用于接收的 characteristic。
2. 如果 characteristic 具备 `notify` 或 `indicate` 能力，则页面调用 `BleService.monitor(serviceUUID, charUUID, onData)`。
3. 每次收到数据时：
   - 保留原始二进制的 Hex
   - 尝试按 UTF-8 转成文本
   - 写入一条 `RECV` 日志
4. 当用户切换监听 characteristic，先停止旧监听，再启动新监听。
5. 页面卸载时调用 `BleService.stopMonitor(serviceUUID, charUUID)` 停止当前调试页监听。

## 状态管理

调试页本地状态包括：

- 当前详情页设备 `deviceId / deviceName`
- 当前连接设备信息
- 已发现的 service 列表
- 当前选中的发送 `serviceUUID / charUUID`
- 当前选中的监听 `serviceUUID / charUUID`
- 待发送消息内容
- `withResponse` 开关状态
- 调试日志列表
- 页面级错误状态

这些状态仅保存在调试页内部，不写入 `AsyncStorage`。

## 错误处理

### 连接状态不匹配

如果 `BleService.getConnectedDevice()` 为空，或与当前详情页设备不一致：

- 页面显示错误提示。
- 通道选择、发送按钮、监听功能全部禁用。
- 不自动重连，也不修改当前 BLE 会话。

### 服务/特征发现失败

- 在日志区追加 `ERROR` 日志。
- 保留错误原文，方便真机联调排查。
- 页面允许用户返回设备详情页，但不尝试兜底重连。

### 发送失败

- 记录 `ERROR` 日志。
- 保留当前输入内容，方便再次尝试。

### 监听失败

- 记录 `ERROR` 日志。
- 停止当前监听选择状态，避免误以为仍在接收。

### 收到不可读二进制数据

- 永远保留 Hex 展示。
- UTF-8 文本解析失败时，只展示 Hex，不把失败视为页面错误。

## UI 与交互约束

- 视觉风格延续现有移动端页面，不引入明显偏 Web 的布局方式。
- 调试页不使用隐藏入口，正式包内持续可见。
- 调试页入口应足够明确，但不要抢占现有主要业务按钮的主语义。
- 日志区优先考虑可滚动、可复制阅读的结构，避免内容被截断后失去调试价值。
- 本次调试相关文案不进入国际化资源，直接以内联中文渲染。

## 风险与边界

### 风险点

- 当前项目连接状态保存在 `BleService` 单例中，调试页依赖它在详情页跳转后仍维持连接会话。
- 不同设备的特征值能力不同，用户可能选择了不可写或不可通知的 characteristic。
- 当前项目页面层尚未广泛使用 `BleService.monitor`，首次接入需保证页面退出时正确停止监听。

### 控制策略

- 不修改连接实现，只消费已有会话。
- 页面层清晰展示每个 characteristic 的能力，减少误选。
- 页面级监听严格在切换与退出时清理，不主动断开连接。

## 验证方案

### 必做验证

- 针对新增/修改文件执行 ESLint。
- 检查设备详情页到调试页的路由跳转是否正确。
- 检查未连接或设备不匹配时，调试页是否正确禁用操作。
- 检查切换监听 characteristic 时，是否停止旧监听。

### 受环境限制的验证

- 无法在当前环境替代真实 BLE 设备做端到端收发验证。
- 真机上需额外验证：
  - 当前设备服务与特征值是否正确列出
  - 发包是否命中目标 characteristic
  - 设备回包是否能通过 notify/indicate 正常接收

## 实施范围总结

这是一次页面层增量功能，不涉及 BLE 核心协议栈调整。实现应保持“小而准”：

- 新增一个独立 BLE 调试页
- 在设备详情页增加入口
- 复用现有 BLE 单例服务
- 不改原生配置，不重构 BLE 核心模块
